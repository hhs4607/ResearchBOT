## 1. Product Overview

ResearchBot is a **review-paper database builder** — a backend-first platform that helps researchers systematically collect, screen, curate, and export academic papers for writing literature reviews.

### 1.1 Core Purpose

Build a curated, high-quality paper database for writing review papers by:
1. Searching across 6 academic APIs with keywords
2. AI-assisted first-pass screening (relevance scoring)
3. User-driven include/exclude decisions via web UI
4. Rich metadata collection (references, cited-by, affiliations, PDF links)
5. Keyword normalization and management
6. Export to CSV and Zotero

### 1.2 Target User

- Individual researcher writing a literature review / review paper
- Needs to search multiple databases, accumulate results over multiple sessions
- Wants AI help for triage but makes final inclusion decisions
- Outputs: curated CSV for analysis, Zotero collection for citation management

### 1.3 What This Is NOT

- Not a real-time paper discovery feed
- Not a batch agent pipeline (no autonomous multi-stage processing)
- Not a note-taking or knowledge management tool (no Obsidian output in v1)
- Frontend implementation is out of scope (handled by Antigravity)

### 2.3 Key Design Principles

1. **Backend-first** — We build the API and DB. Frontend is a separate concern.
2. **Project-based accumulation** — Multiple searches accumulate into one project DB. No ephemeral sessions.
3. **User-driven curation** — AI screens, user decides. Include/exclude is always a human choice (with optional auto-select threshold).
4. **Maximum metadata** — Collect everything available: refs, cited-by, affiliations, PDF links. User can decide later what matters.
5. **Keyword integrity** — Normalized, deduplicated keywords are critical for review paper quality.
6. **Best-effort sources** — Google Scholar (scraping) may fail; other sources continue. Never skip silently without logging.

## 3. Data Model

### 3.1 Database Schema (8 tables)

```sql
-- Projects: a review paper = a project
CREATE TABLE projects (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,                    -- e.g., "PINN Fatigue Review 2026"
    description TEXT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Papers: unified paper metadata from all sources
CREATE TABLE papers (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL REFERENCES projects(id),

    -- Identifiers (at least one required)
    doi TEXT,
    openalex_id TEXT,
    s2_id TEXT,
    arxiv_id TEXT,

    -- Core metadata
    title TEXT NOT NULL,
    authors TEXT,                          -- JSON array: [{name, affiliation}]
    year INTEGER,
    venue TEXT,                            -- journal or conference name
    abstract TEXT,
    url TEXT,                              -- canonical paper URL
    pdf_url TEXT,                          -- direct PDF link if available
    tldr TEXT,                             -- short summary (from S2 or LLM)

    -- Rich metadata
    cited_by_count INTEGER DEFAULT 0,
    is_open_access BOOLEAN DEFAULT 0,
    references TEXT,                       -- JSON array: [{doi?, title, year?}]
    cited_by TEXT,                         -- JSON array: [{doi?, title, year?}]

    -- AI-generated fields
    ai_relevance_score REAL,              -- composite screening score (0.0-1.0)
    ai_keywords TEXT,                      -- semicolon-separated AI-extracted keywords
    ai_objective TEXT,                     -- research objective (1 sentence)
    ai_method TEXT,                        -- methodology (1 sentence)
    ai_result TEXT,                        -- key findings (1 sentence)

    -- User curation
    is_included INTEGER,                   -- NULL=undecided, 1=include, 0=exclude
    user_note TEXT,                         -- optional user memo

    -- Source tracking
    sources TEXT,                          -- JSON array: ["openalex","s2","scholar"]
    raw_metadata TEXT,                     -- JSON: full API responses merged

    -- Timestamps
    discovered_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    -- Uniqueness within a project
    UNIQUE(project_id, doi),
    UNIQUE(project_id, title)
);

-- Keywords: normalized keyword dictionary
CREATE TABLE keywords (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    canonical_form TEXT NOT NULL UNIQUE,    -- e.g., "PINN"
    variants TEXT,                          -- JSON array: ["physics-informed neural network", "physics informed NN"]
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Paper <-> Keyword junction
CREATE TABLE paper_keywords (
    paper_id INTEGER NOT NULL REFERENCES papers(id) ON DELETE CASCADE,
    keyword_id INTEGER NOT NULL REFERENCES keywords(id) ON DELETE CASCADE,
    source TEXT DEFAULT 'ai',              -- 'ai' | 'author' | 'user'
    PRIMARY KEY (paper_id, keyword_id)
);

-- Search logs: track what searches were performed
CREATE TABLE search_logs (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL REFERENCES projects(id),
    query TEXT NOT NULL,                    -- the search keywords used
    search_mode TEXT DEFAULT 'standard',   -- quick|standard|deep
    source_results TEXT,                   -- JSON: {"openalex": 15, "s2": 12, ...}
    papers_found INTEGER DEFAULT 0,        -- total results before dedup
    papers_new INTEGER DEFAULT 0,          -- actually new papers added
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- Search <-> Paper junction (which search found which paper)
CREATE TABLE search_papers (
    search_id INTEGER NOT NULL REFERENCES search_logs(id) ON DELETE CASCADE,
    paper_id INTEGER NOT NULL REFERENCES papers(id) ON DELETE CASCADE,
    PRIMARY KEY (search_id, paper_id)
);

-- Paper source details: per-source metadata
CREATE TABLE paper_sources (
    paper_id INTEGER NOT NULL REFERENCES papers(id) ON DELETE CASCADE,
    source TEXT NOT NULL,                  -- openalex|s2|arxiv|crossref|pubmed|scholar
    source_id TEXT,                        -- source-specific ID
    metadata TEXT,                         -- JSON: source-specific extra data
    fetched_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (paper_id, source)
);

-- Zotero sync tracking
CREATE TABLE zotero_sync (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    project_id INTEGER NOT NULL REFERENCES projects(id),
    collection_key TEXT,                   -- Zotero collection ID
    papers_synced INTEGER DEFAULT 0,
    last_synced_at TIMESTAMP
);
```

### 3.2 Deduplication Strategy

Papers are deduplicated within a project by:
1. **DOI match** — exact (primary)
2. **Title fuzzy match** — rapidfuzz ratio >= 90 (secondary)
3. **Cross-source ID match** — same openalex_id, s2_id, or arxiv_id

When a duplicate is found, metadata is **merged** (take the richest version of each field).

## 5. Features

### 5.1 Project Management

- Create/rename/delete review projects
- Each project has its own paper collection
- Project-level settings (auto-select threshold, default search mode)

### 5.2 Search & Collection

**Core workflow**:
1. User enters keywords → system searches 6 APIs → results displayed with composite scores
2. Results are **temporary** (in-memory, not saved to DB)
3. User selects papers to save (checkbox) → selected papers persist to project DB
4. User triggers Gemini extraction on saved papers (separate step)
5. Accumulated papers across multiple searches form the project database

```
Keywords → [6 API Search] → [Dedup + Score] → Temporary Results
                                                    ↓
                                          User selects papers
                                                    ↓
                                    POST /papers/save → DB
                                                    ↓
                                    POST /papers/extract → Gemini (keywords + OMR)
```

**From 07_Research_Bot (port):**
- Multi-source parallel search with query rewriting (Gemini)
- 7-signal composite relevance scoring
- Score optimization with penalty transform
- 66+ domain-specific acronym expansion
- Search modes: quick (OpenAlex only) / standard (3 sources) / deep (all 6)

**From PaperReviewBot (port):**
- CrossRef, PubMed, Google Scholar clients
- Additional acronym mappings (merge with existing → deduplicated union)
- DOI + fuzzy title deduplication (rapidfuzz)

**New:**
- Search results are temporary — only explicitly saved papers enter the DB
- Search history tracked in search_logs (what was searched, not what was saved)
- Auto-select option: on saved papers, `ai_relevance_score >= threshold` → auto-include
- Cross-project paper copy

### 5.3 AI Screening (Gemini)

**Two separate stages:**

**Stage 1 — Composite scoring (during search, fast, no LLM):**
- Compute composite relevance score (0.0–1.0) using 7-signal scoring
- Applied to all search results before display

**Stage 2 — Gemini extraction (after save, on-demand):**
- Extract keywords, objective, method, result via Gemini
- Triggered per-paper or in bulk on saved papers
- Results stored in ai_keywords (raw), ai_objective, ai_method, ai_result
- Extracted keywords auto-normalized against keyword dictionary → paper_keywords table

**Scoring signals** (from 07_Research_Bot Phase 3 optimized config):

| Signal | Weight | Description |
|--------|--------|-------------|
| text_match | 0.30 | Keyword matching with acronym expansion |
| relevance | 0.20 | API-provided relevance score |
| citations | 0.15 | Log-normalized citation count |
| multi_source | 0.15 | Found in multiple APIs |
| recency | 0.10 | Year-based linear decay |
| abstract | 0.10 | Has abstract/TLDR |
| semantic | 0.00 | Reserved for future SPECTER2 |

Plus penalty transform: `text_match < 0.30 → score × 0.50`

### 5.4 Paper Database Management (Core Feature)

This is the **most important screen**. The API must support:

**Listing & Filtering:**
- List all papers in a project with pagination
- Filter by: keyword, year range, include/exclude status, relevance score range, source
- Sort by: relevance score, year, citations, title, discovered_at
- Full-text search within titles and abstracts

**Individual Paper Operations:**
- View full paper details (all metadata fields)
- Toggle include/exclude status
- Edit AI-extracted keywords (add/remove/modify)
- Edit classification/tags
- Edit AI-extracted objective/method/result
- Add user notes/memo
- Re-fetch metadata from sources (enrich missing fields)

**Bulk Operations:**
- Select multiple papers → bulk include/exclude
- Auto-select: mark all papers above score threshold as included
- Bulk keyword assignment

### 5.5 Keyword Management

**Normalization:**
- Canonical keyword dictionary with variant mappings
- Example: canonical "PINN" ← variants ["physics-informed neural network", "physics informed NN", "PINN"]
- When AI extracts keywords, auto-map to canonical forms
- User can add new canonical keywords and variants at any time

**From PaperReviewBot (port):**
- 25+ abbreviation mappings (ABBREVIATION_MAP in pipeline/keywords.py)
- Keyword frequency computation

**From 07_Research_Bot (port):**
- 66+ domain-specific acronym expansions (intent_parser.py)
- Merge both into single unified dictionary

### 5.6 Export

**CSV Export:**
- Export included papers as CSV
- Configurable columns (user selects which fields to include)
- Default columns: title, first_author, doi, journal, year, keywords, objective, method, result

**Zotero Sync:**
- Create Zotero collection per project
- Push included papers by DOI
- Track sync status per project

### 5.7 Chrome Crawling (Future — Phase 2)

For papers that API sources miss or return incomplete metadata:
- Browser-based crawling to supplement missing data
- Google Scholar full coverage via browser automation
- Specific publisher page scraping for additional metadata
- Implementation approach TBD (Playwright, Selenium, or Chrome Extension)

## 6. API Design (REST Endpoints)

### 6.1 Projects

```
GET    /api/projects                    List all projects
POST   /api/projects                    Create project
GET    /api/projects/{id}               Get project details
PUT    /api/projects/{id}               Update project
DELETE /api/projects/{id}               Delete project + all papers
```

### 6.2 Search

```
POST   /api/projects/{id}/search        Execute search (returns TEMPORARY results, no DB save)
GET    /api/projects/{id}/searches       List search history
POST   /api/projects/{id}/auto-select    Auto-include saved papers above threshold
```

### 6.3 Papers

```
# Save from search results
POST   /api/projects/{id}/papers/save    Save selected papers from search to DB

# CRUD
GET    /api/projects/{id}/papers         List saved papers (filter, sort, paginate)
GET    /api/papers/{id}                  Get paper detail (full metadata)
PUT    /api/papers/{id}                  Update paper (keywords, note, OMR, etc.)
PATCH  /api/papers/{id}/include          Toggle include/exclude
DELETE /api/papers/{id}                  Remove paper from project

# AI extraction (Gemini) — separate from search
POST   /api/papers/{id}/extract          Trigger Gemini extraction for one paper
POST   /api/projects/{id}/papers/extract Bulk Gemini extraction

# Enrichment
POST   /api/papers/{id}/refetch          Re-fetch metadata + refs/cited_by from sources
POST   /api/papers/{id}/copy             Copy paper to another project

# Bulk operations
POST   /api/projects/{id}/papers/bulk-include    Bulk set include/exclude
POST   /api/projects/{id}/papers/bulk-keywords   Bulk assign keywords
```

### 6.4 Keywords

```
GET    /api/keywords                     List all canonical keywords
POST   /api/keywords                     Create keyword (canonical + variants)
PUT    /api/keywords/{id}                Update keyword (add/remove variants)
DELETE /api/keywords/{id}                Delete keyword
GET    /api/projects/{id}/keyword-stats  Keyword frequency in project
```

### 6.5 Export

```
GET    /api/projects/{id}/export/csv     Export as CSV (query params for column selection)
POST   /api/projects/{id}/export/zotero  Sync included papers to Zotero
GET    /api/projects/{id}/export/zotero/status  Check sync status
```

## 11. Decisions Log

| Decision | Rationale |
|----------|-----------|
| Web-only (no Agent pipeline) | Goal is interactive curation, not batch processing |
| Project-based, not session-based | Review papers need accumulated DBs, not one-shot searches |
| Gemini (not Claude) for LLM | Consistent with 07_Research_Bot; cheaper for bulk screening |
| httpx everywhere (not requests) | Async-ready, consistent with FastAPI |
| 07_RB search clients as base | Standalone modules, richer metadata, already httpx-based |
| Backend-only scope | Frontend by Antigravity; we provide REST API |
| 8 tables (not 15) | Removed agent pipeline tables; kept what's needed for curation |
| CSV + Zotero export (no Obsidian) | Focused on review paper workflow outputs |
| Keyword normalization as core | Critical for review paper quality; prevents "PINN" vs "physics-informed neural network" fragmentation |
| **Search results are temporary** | Search returns in-memory results. Only user-selected papers are saved to DB. Prevents DB bloat from exploratory searches. |
| **Gemini extraction deferred** | NOT called during search. Triggered separately on saved papers. Saves cost (+speed), user only pays for papers they care about. |
| **ai_keywords vs paper_keywords** | ai_keywords = Gemini raw output (read-only archive). paper_keywords = operational normalized data (user edits this). |
| **refs/cited_by on refetch only** | Not collected during search (too slow). User triggers refetch to enrich a saved paper. |
| **Cross-project paper copy** | Papers can be copied between projects. Copy creates new row with metadata duplicated, is_included reset to null. |
